# 遇见新疆 uni-app 项目结构梳理

## 1. 项目定位

- 这是一个以 `uni-app + Vue 3` 为前端壳的安卓项目，主题是新疆旅游服务。
- 前端主要包含景区浏览、攻略信息流、攻略发布、AI 助手、账号中心、徒步模式、安全地图等模块。
- 项目同时内置了一套 `Node.js + Express + PostgreSQL` 的后端服务，用于认证、景区数据、攻略数据、媒体上传等能力。
- 当前工程既支持“本地 mock 数据直接运行”，也支持切换到远程接口模式。

## 3. 前端入口与全局配置

### `main.js`

- 使用 `createSSRApp(App)` 创建 uni-app 应用实例。
- 项目当前没有额外注册全局插件，结构比较轻。

### `App.vue`

- 全局生命周期入口。
- `onHide` / `onUnload` 时会清空 AI 会话缓存。
- 引入全局样式 `common/common.scss`。

### `pages.json`

- 定义了整个应用的页面路由。
- 当前已注册的主要页面：
  - `pages/home/index` 首页
  - `pages/destinations/index` 目的地列表
  - `pages/destination-detail/index` 景区详情
  - `pages/guides/index` 攻略信息流
  - `pages/guide-detail/index` 攻略详情
  - `pages/guide-publish/index` 发布攻略
  - `pages/ai-assistant/index` AI 助手
  - `pages/account/index` 个人中心
  - `pages/auth/index` 登录注册
  - `pages/safety-map/index` 安全地图
  - `pages/hiking/index` 徒步模式
- 页面大多使用 `navigationStyle: custom`，说明导航栏主要是自定义实现。

### `manifest.json`

- 当前是 Android App 配置中心。
- 已申请网络、定位等权限。
- `appid` 还是占位值，正式打包前需要替换。

### `uni.scss` + `common/common.scss`

- 定义了整个项目统一的视觉变量：主色、辅助色、背景色、文字色、边框色。
- `common/common.scss` 里封装了页面壳、卡片、渐变头图、通用间距等基础样式。
- 当前 UI 风格整体偏暖色旅游内容风格。

## 4. 主要目录说明

### `pages/`

这是业务页面主体，当前页面职责大致如下：

- `home/`：首页，展示景区推荐、附近景区、热门玩法，并可跳转 AI 助手、徒步模式。
- `destinations/`：景区列表页，支持搜索、分类筛选、地区筛选。
- `destination-detail/`：景区详情页，集成地图、路线估算、实时天气、离线安全图、AI 建议、抖音搜索入口。
- `guides/`：攻略发现页，采用双列瀑布流样式，支持分类切换和发布入口。
- `guide-detail/`：攻略详情页，支持图文/视频/纯文字展示。
- `guide-publish/`：发布攻略，支持图片、视频、位置、标签，并可落到本地或服务端。
- `ai-assistant/`：AI 旅游助手，接入阿里百炼兼容接口，支持上下文问答。
- `account/`：个人中心，展示用户信息、头像上传、我的攻略、账户设置、退出登录。
- `auth/`：登录注册页，对接认证接口。
- `safety-map/`：景区安全地图页，可展示在线或离线版本。
- `hiking/`：徒步模式页，包含定位、离线图、轨迹、紧急联系人、求救按钮、模拟户外安全场景。
- `live-webview/`：内嵌 WebView，用于打开直播搜索页。

### `components/`

- `AppTabBar.vue`：底部主导航栏组件，贯穿首页、目的地、攻略、AI、我的五个一级入口。
- `CachedImage.vue`：图片展示组件，用于统一封装图片加载逻辑。

### `common/`

这里更多是“本地数据 + 本地状态管理 + 工具逻辑”：

- `destination-data.js`：大型景区基础数据仓，包含景区列表、文化信息、旅行元信息、安全图信息、直播搜索词等。
- `guide-data.js`：本地攻略 mock 数据。
- `published-guides.js`：用户发布攻略的本地持久化逻辑。
- `auth-storage.js`：登录 token 和用户信息的本地缓存。
- `offline-map.js`：离线地图下载、保存、读取、删除。
- `hiking-session.js`：徒步过程中的定位轨迹缓存。
- `hiking-mode.js`：徒步模式 mock 数据。

### `services/`

这里是前端和配置/接口之间的桥梁层：

- `destinations.js`：景区列表/详情服务，优先走远程接口，失败后回退本地数据。
- `guides.js`：攻略流、攻略详情、媒体上传、攻略发布服务，支持接口模式和本地模式双通道。
- `auth.js`：登录、注册、头像上传、资料更新、我的攻略。
- `amap.js`：高德地图能力封装，包括静态地图、逆地理编码、实时天气、路线规划、获取定位。
- `ai.js`：AI 助手服务，负责拼接景区和攻略上下文并请求大模型接口。

### `config/`

- `api.js`：内容服务 API 地址。
- `auth.js`：认证相关 API 地址与本地存储 key。
- `amap.js`：高德 Web Key。
- `ai.js`：AI 模型、接口地址、API Key 存储逻辑。

### `server/`

这是内置的后端服务目录：

- `index.js`：Express 主服务，提供健康检查、注册登录、景区列表、景区详情、攻略列表、攻略详情、用户资料、头像上传、攻略发布、媒体上传等接口。
- `db.js`：PostgreSQL 连接池封装。
- `config.js`：端口、JWT 密钥、数据库配置。
- 其他迁移脚本：`migrate.js`、`migrate-guide-notes.js`、`migrate-author-id.js` 等。

### `docs/`

- 存放 PostgreSQL 建表、种子数据、认证表结构、攻略迁移等 SQL/说明文档。
- 如果后续要本地部署后端，这个目录是数据库初始化的重要入口。

### `static/`

- 静态资源目录，主要用于图标、图片等素材。

## 5. 运行模式理解

这个项目目前实际上有两套数据来源：

### 本地 mock 模式

- 景区从 `common/destination-data.js` 读取。
- 攻略从 `common/guide-data.js` 读取。
- 用户发布的攻略写入本地存储 `published-guides.js`。
- 离线地图、登录态、徒步轨迹也都保存在本地存储。

### 远程 API 模式

- 当 `config/api.js` 或 `config/auth.js` 中配置了有效地址时，前端优先请求远程接口。
- 如果接口失败，大部分页面都做了本地 fallback，不至于直接白屏。
- 这是一个“先用本地数据把壳搭好，再逐步联调后端”的典型结构。

## 6. 关键业务流

### 景区流

`首页/目的地列表 -> 景区详情 -> 地图/天气/安全图/AI 推荐`

景区详情页是全项目里最重的页面之一，聚合了：

- 景区介绍
- 高德地图静态图
- 驾车/步行路线估算
- 实时天气
- 离线安全图下载
- AI 行程建议
- 抖音直播搜索入口

### 攻略流

`攻略信息流 -> 攻略详情 -> 发布攻略 -> 回流信息流/个人中心`

- 攻略支持图文、视频、纯文字三种形式。
- 发布时会根据标题/内容自动推断子分类，如自驾、美食、安全、徒步、住宿。
- 如果接了远程接口，会上传媒体并写入数据库；否则走本地存储。

### 用户流

`登录注册 -> 本地保存 token/user -> 个人中心 -> 更新昵称/头像 -> 查看我的攻略`

- 登录态由 `common/auth-storage.js` 管理。
- 个人中心会调用 `/users/me`、`/users/me/avatar`、`/users/me/guides` 等接口。

### 户外安全流

`景区详情安全卡 -> 安全地图 -> 离线图下载 -> 徒步模式`

- 项目对“离线地图 + GPS + 安全提醒”做了较完整的产品原型。
- `hiking` 页目前更多是高保真演示页，核心是真机定位与本地缓存，部分安全能力还是演示 toast。

## 7. 后端接口大致能力

根据 `server/index.js`，当前后端已具备这些接口能力：

- `GET /api/health`
- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/destinations`
- `GET /api/destinations/:id`
- `GET /api/guides`
- `GET /api/guides/:slug`
- `POST /api/guides/media`
- `POST /api/guides`
- `PATCH /api/users/me`
- `POST /api/users/me/avatar`
- `GET /api/users/me/guides`
- `POST /api/users/me/guides/:slug/claim`

整体上已经不是单纯的 UI 工程，而是“前端 + 简单内容后端 + 数据库脚本”一体项目。

## 8. 当前项目特点

- 页面完成度较高，尤其是景区详情、攻略流、AI 助手、徒步模式四块。
- 代码组织比较清晰：`pages` 管界面，`services` 管请求，`common` 管本地数据和本地状态。
- 很多能力都做了 fallback，适合边开发边联调。
- 项目明显偏移动端体验，尤其是自定义导航、底部 Tab、离线能力、定位能力。

## 9. 我目前观察到的注意点

- `server/config.js` 中写死了数据库地址、账号、密码、JWT secret，后续正式部署前建议改为环境变量。
- `config/amap.js` 里已经存在高德 Key，正式发布前建议确认是否需要更换或限制来源。
- `config/api.js` 当前指向固定公网地址，开发时要确认这个地址是否仍有效。
- 目录里包含 `node_modules/` 和 `unpackage/`，后续如果做仓库清理，可以考虑是否需要忽略生成物。

这个项目不是单纯安卓壳，而是一个已经做出完整旅游产品雏形的 `uni-app` 工程：前端页面丰富，数据既能本地 mock 也能走远程接口，还内置了认证、攻略发布、媒体上传、离线地图和 AI 助手能力。
