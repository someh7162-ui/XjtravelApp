# 云起天山 APK 无法连接后端的具体原因

## 结论

当前问题**不是后端 Node 服务没启动**，也**不是接口路径写错了**，核心原因是：

1. 你的后端对外暴露地址 `https://yd.frp-arm.com:44637` 使用的是 **SakuraFrp 自签名证书**。
2. Android 里的 `uni.request`、`uni.downloadFile`、`<image>`、WebView/Chromium、部分 DCloud 内部网络能力，**并不一定都走同一套你手工放进去的证书信任链**。
3. 所以会出现一种情况：
   - 某些请求能通
   - 某些请求、图片、头像、资源下载仍然因为 **SSL 校验失败** 而报错
4. 攻略页依赖的不只是 `/api/guides` 列表接口，还依赖图片、头像、上传、详情等多个 HTTPS 资源；只要其中一部分网络栈不认这个证书，页面就会表现成“还是连不上”。

## 已确认的事实

### 1. 后端本身是活着的

之前已确认服务器本机接口可用：

- `http://127.0.0.1:3000/api/guides` 可访问

这说明：

- Node/Express 服务在运行
- 路由 `/api/guides` 存在
- 不是数据库挂了导致完全不可用

### 2. 外网 HTTPS 地址只有在“忽略证书校验”或“显式信任自签证书”时才可访问

本地测试结果已经证明：

- 访问 `https://yd.frp-arm.com:44637/api/guides`
- 使用系统默认证书校验时：**失败**
- 使用嵌入的 `sakura_frp_yd_cert.pem` 时：**成功返回 200**

这说明：

- 域名和服务是通的
- 真正卡住的是 **证书信任问题**，不是“服务器没开”

### 3. HTTP 也不能直接拿来替代

测试 `http://yd.frp-arm.com:44637/api/guides` 返回 `501 Not Implemented`。

这说明当前穿透/转发配置不是一个可直接给 App 用的普通 HTTP 入口，不能简单把前端地址改成 `http://...` 就解决。

### 4. 离线 APK 里最开始并没有完整地图/高德模块

这个问题我已经修过，但它和“攻略无法连后端”是两件事。

地图模块缺失会影响徒步页 `<map>`，**不会直接导致攻略接口完全不可达**。

## 为什么 App 里比命令行更容易失败

这是当前最关键的一点。

你的测试环境里，至少同时存在下面几种网络通路：

1. `uni.request`
2. `uni.uploadFile`
3. `uni.downloadFile`
4. `<image>` 标签直接加载远程图片
5. WebView / Chromium 内核加载资源
6. DCloud/5+ 原生网络能力

这些能力**不一定共享同一个证书信任配置**。

也就是说，即使我在 Android 工程里加了：

- `network_security_config.xml`
- `@raw/sakura_frp_yd_cert`

也不能保证 **所有** 资源请求都完全接受这张自签名证书。

这就是为什么会出现：

- 接口探测有时能通
- 页面仍然报“连接失败”
- 图片/头像可能继续失败
- Chromium 日志里之前出现过 SSL handshake 错误

## 现在这个问题的真正根因

可以明确归纳为下面两条：

### 根因 1：外网入口使用自签名 HTTPS

`https://yd.frp-arm.com:44637` 的证书不是公开 CA 证书，而是自签名证书。

结果：

- Android 默认不信任
- uni-app / DCloud / WebView 某些请求链路不会稳定接受它
- 只要有一个页面关键请求走到了不信任这张证书的链路，就会表现成“连不上后端”

### 根因 2：攻略页不仅请求 JSON，还请求图片和头像等静态资源

攻略列表接口返回的数据里包含：

- 封面图
- 多图资源
- 作者头像
- 视频/视频封面

这些 URL 虽然我已经做过域名归一化，但它们仍然走 HTTPS 下载。

因此只要：

- JSON 请求成功
- 但图片/头像下载失败

页面仍然可能被用户感知为“攻略加载失败”或者“看起来没正常连上”。

## 为什么这不是前端 API_BASE_URL 配错

当前前端主地址已经统一为：

- `https://yd.frp-arm.com:44637`

并且旧地址：

- `https://111.20.31.227:34144`
- `https://frp-arm.com:44637`

也已经做了归一化处理。

所以当前不是简单的“前端写错服务器地址”。

## 对测试分发来说，最实际的解决方案

既然你明确说了：

- 现在不追求安全
- 只想先打包分发给一部分人测试

那最实际的方案排序如下。

### 方案 A（最推荐）：给穿透域名换成公网受信任证书

例如：

- 用 Nginx / Caddy / 宝塔 / 反代
- 给 `yd.frp-arm.com` 配 Let’s Encrypt 证书

优点：

- App、WebView、图片、上传、下载基本都会一起恢复正常
- 不需要继续和 uni-app / DCloud / Android 多套网络栈对抗
- 对外分发最稳

### 方案 B：换成一个真正可用的明文 HTTP 测试入口

前提是你的穿透服务支持普通 HTTP，不要返回 501。

优点：

- 对测试最粗暴有效
- 可以彻底绕开自签证书问题

缺点：

- 现在你的入口还不能这么用，因为 `http://yd.frp-arm.com:44637/api/guides` 当前就是 501

### 方案 C：继续在 App 里强行绕过证书

我已经开始往这个方向做了：

- 原生请求封装
- 原生下载封装
- 缓存图片下载改走 `plus.downloader`

但这个方案的问题是：

- 需要逐个排查所有请求入口
- 只要还有一个地方落回 WebView/Chromium 或 uni 内部默认网络栈，还是会失败
- 对“分发给别人测”不如服务器侧换证书稳

## 一句话判断

**当前 App 连不上你后端的最具体原因，就是：你的外网后端入口用了自签名 HTTPS，而 uni-app/DCloud/Android/WebView 的多套网络与资源加载链路不能稳定、统一地信任这张证书。**

这不是后端没启动，也不是单纯接口地址写错，而是**测试分发环境下的证书信任链问题**。

## 下一步建议

如果目标是“尽快给一批人装包测试”，优先级建议这样排：

1. **先把外网入口换成公开 CA 证书**（最稳）
2. 如果做不到，就**提供真正可用的 HTTP 测试入口**
3. 如果服务器侧暂时完全不能改，再继续在 App 里逐项兜底所有 HTTPS 请求链路
