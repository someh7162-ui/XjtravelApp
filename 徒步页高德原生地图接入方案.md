# 徒步页高德原生地图接入方案

## 目标

在 `uni-app` 项目的 `pages/hiking/index.vue` 中接入更适合徒步安全场景的地图能力，实现：

1. 真实地图显示，而不是占位图
2. 真实定位、海拔、精度、速度、里程记录
3. 支持更适合户外场景的地图模式切换
4. 为后续偏航提醒、危险区域提示、离线记录、SOS 提供基础能力

当前建议采用：

- `uni-app` 页面负责 UI
- Android 原生高德地图 SDK 负责地图、定位、轨迹、图层能力
- uni-app 通过原生插件或 `plus.android` 与 Android 通信

不建议只依赖 `uni-app` 默认 `<map>` 组件来做最终徒步方案，因为它更适合基础地图展示，不适合做专业户外安全地图。

## 推荐方案

### 方案选型

采用下面的结构：

- 页面层：`pages/hiking/index.vue`
- 原生地图层：Android 高德地图 SDK
- 通信层：`uni.requireNativePlugin(...)` 优先，`plus.android` 作为备选
- 数据层：原生侧持续定位并回传实时数据

## 为什么不用纯 `<map>`

纯 `uni-app` `<map>` 的问题：

1. 图层能力有限
2. 难以做更专业的地形/卫星/户外样式切换
3. 对连续轨迹记录、定位精度控制、原生事件回传支持较弱
4. 后续离线、安全提醒、后台持续定位扩展成本高

所以徒步页最终应改成：

- UI 还是 `vue`
- 地图内核换成 Android 原生高德地图 SDK

## 地图能力目标

徒步页地图区域建议支持以下能力：

### 基础能力

- 显示当前真实位置
- 跟随当前位置移动
- 绘制徒步轨迹折线
- 显示起点、当前位置、补给点、危险点
- 支持手动刷新定位
- 支持地图回中

### 徒步安全能力

- 卫星/普通/户外增强模式切换
- 精度过低提醒
- 长时间静止提醒
- 偏离路线提醒
- 弱网下继续本地轨迹记录
- SOS 时附带当前位置、海拔、最近轨迹点

### 数据面板能力

- 海拔：来自原生定位 altitude
- 里程：由原生连续轨迹点累计
- 精度：来自 accuracy
- 速度：来自 speed
- 坐标：当前经纬度

## 技术架构

### 1. uni-app 页面层

页面文件：`pages/hiking/index.vue`

负责：

- 顶部状态 UI
- 底部控制 UI
- 模式切换 UI
- SOS 和守护模式 UI
- 接收原生数据并展示

页面不再自己承担核心地图逻辑，只负责调原生能力。

### 2. Android 原生层

建议新增一个原生地图插件，内部封装高德 Android SDK。

原生层负责：

- 初始化地图
- 持续定位
- 轨迹点存储
- 里程累计
- 海拔读取
- 图层切换
- 事件回传给 uni-app

### 3. 通信层

推荐优先级：

1. `uni.requireNativePlugin('插件名')`
2. `plus.android` 直接调用原生类

如果是长期维护方案，推荐做正式原生插件，不建议把所有逻辑都堆在 `plus.android` 调用里。

## 原生插件接口设计

建议原生插件暴露这些方法：

### 初始化

- `initMap(options)`
- `destroyMap()`

### 地图控制

- `setMapMode(mode)`
  - `normal`
  - `satellite`
  - `terrain-like`
- `moveToCurrentLocation()`
- `setCompassEnabled(enabled)`
- `setZoom(level)`

### 定位与轨迹

- `startLocation()`
- `stopLocation()`
- `startTrack()`
- `stopTrack()`
- `clearTrack()`
- `getTrackSummary()`

### 覆盖物

- `setRoutePolyline(points)`
- `setDangerMarkers(markers)`
- `setSupplyMarkers(markers)`

### 事件

原生侧应持续向前端回传：

- `onLocationChange`
- `onTrackUpdate`
- `onGpsWeak`
- `onDeviationAlert`

## 徒步页前端改造方案

### 现阶段页面改造思路

`pages/hiking/index.vue` 改为三块：

1. 顶部数据面板
2. 中部原生地图容器
3. 底部操作区

### 页面需要保留的前端状态

- `isTracking`
- `isGuardMode`
- `locationInfo`
- `trackSummary`
- `gpsStatus`
- `currentMapMode`

### 页面不建议继续自己做的逻辑

- 用 JS 轮询算总里程
- 用前端假数据填海拔
- 用静态图片模拟地图

这些都应迁移到原生层。

## 地形图问题说明

这里要特别说明：

### 1. 高德是否直接提供“徒步地形图”

在 `uni-app` 默认 `<map>` 层面，基本不能直接拿到专业户外地形图体验。

在 Android 原生高德 SDK 层面，可以更接近以下效果：

- 普通地图
- 卫星图
- 卫星路网叠加
- 更适合户外的图层切换能力

但“真正专业等高线地形图”是否完全可用，还要看高德 Android SDK 当前开放能力。

### 2. 如果高德地形表达不够

可以采用折中方案：

- 高德 SDK 负责定位、地图交互、标记、轨迹
- 叠加自定义风险图层/路线图层
- 后续如有需要，再评估接入专业地形瓦片服务

所以第一阶段先做：

- 高德原生地图 + 卫星/普通模式 + 轨迹 + 安全提醒

这已经足够明显提升徒步安全性。

## 与当前项目的衔接点

当前项目中已有可复用基础：

- `services/amap.js`
  - 当前可用于高德坐标定位和基础 Web 服务调用
- `common/hiking-session.js`
  - 当前可作为前端本地缓存兜底
- `pages/hiking/index.vue`
  - 当前作为徒步 UI 页面继续保留

后续建议：

- `services/amap.js` 保留给普通 Web 服务能力
- 徒步页实时地图和轨迹改走 Android 原生插件

## 实施步骤

### 第 1 步：补文档和接口约定

- 明确原生插件名称
- 明确方法名
- 明确事件名
- 明确回传 JSON 结构

### 第 2 步：Android 侧接入高德 SDK

- 引入高德 Android 地图 SDK
- 创建原生地图容器
- 打通定位与轨迹记录

### 第 3 步：uni-app 页面接入原生插件

- `pages/hiking/index.vue` 调 `initMap`
- 启动/停止轨迹记录
- 监听原生回传并更新顶部面板

### 第 4 步：加入安全能力

- 偏航提醒
- 危险区域覆盖物
- SOS 真实位置回传

## 风险与注意事项

### 1. 平台差异

优先先做 Android，iOS 可以后补。

### 2. 权限

需要继续确认 Android 权限完整性，可能还需要：

- 后台定位相关权限
- 前台服务权限

### 3. 电量消耗

徒步模式连续定位会耗电，需要增加：

- 省电模式
- 高精度模式
- 定位频率可配置

### 4. 弱网与离线

徒步页不能依赖实时网络，轨迹和关键状态必须本地可用。

## 结论

对于 `uni-app` 徒步页，最佳方案不是继续只用默认 `<map>`，而是：

- `uni-app` 做 UI
- Android 原生高德 SDK 做地图和定位核心
- 原生侧负责真实海拔、真实里程、真实轨迹

这样才能真正做出适合徒步安全场景的地图页。
