# 攻略页与离线打包修复记录

## 当前结论

- 目前模拟器中攻略内容已经可以显示，说明本轮修复基本生效。
- 这次问题的核心不是单一接口失败，而是离线 Android 包、旧资源地址、自签名 HTTPS、WebView 直连资源这几类问题叠加。
- 其中最关键的一点是：后端返回的数据里仍可能包含旧资源地址 `111.20.31.227:34144`，所以前端必须做兼容改写，不能只依赖后端完全修正。

## 这次做过的主要修改

### 1. 统一攻略接口入口

文件：`config/api.js`

- 将 API 入口统一到：`https://yd.frp-arm.com:44637`
- 保留并兼容旧地址改写：
  - `https://111.20.31.227:34144`
  - `http://111.20.31.227:34144`
  - `https://frp-arm.com:44637`
  - `http://frp-arm.com:44637`
- 新增了对“不带协议旧地址”的兼容，例如：
  - `111.20.31.227:34144/uploads/...`
  - `frp-arm.com:44637/uploads/...`
- 通过 `normalizeApiAssetUrl()` 将旧资源地址统一重写到新入口。

### 2. 新增原生 HTTPS 请求与下载封装

文件：`common/app-http.js`

- 新增 `requestJson()`：
  - App 端优先走 `plus.net.XMLHttpRequest`
  - 其他端回退到 `uni.request`
- 新增 `downloadRemoteFile()`：
  - App 端优先走 `plus.downloader`
  - 其他端回退到 `uni.downloadFile`
- 新增 `normalizeLegacyUrl()` 与 `normalizePayloadUrls()`：
  - 对接口返回的对象、数组、字符串递归处理
  - 自动把旧 IP / 旧域名资源改写成新地址
- 目的：
  - 避免 Android WebView 或 uni 默认请求链路直接碰自签名 HTTPS
  - 避免旧资源地址继续打到 `111.20.31.227:34144`

### 3. 攻略服务改成走统一请求封装

文件：`services/guides.js`

- 原来的攻略请求改为走 `requestJson()`
- 新增 `normalizeGuideEntity()`，统一处理攻略实体中的这些字段：
  - `image`
  - `images`
  - `authorAvatar`
  - `video`
  - `videoPoster`
- 评论接口也增加了头像地址标准化：
  - `avatarUrl`
  - `authorAvatar`
- 目的：
  - 无论接口返回什么旧地址，页面使用前都会再做一次修正。

### 4. 认证服务改成走统一请求封装

文件：`services/auth.js`

- 登录、注册、用户资料、关注等请求改为统一走 `requestJson()`
- 保持请求头与 token 逻辑一致
- 作用：
  - 减少认证相关请求在 Android 端因 HTTPS 差异导致的不一致问题。

### 5. 图片组件改为优先下载本地缓存，避免 WebView 直接访问远程 HTTPS

文件：`components/CachedImage.vue`

- 新增 `fallbackToRemote` 参数
- App 端对于远程图片：
  - 先查本地缓存
  - 没缓存则先下载
  - 下载成功后使用本地文件路径显示
- 在原生 App 中默认不再随便回退到远程 HTTPS 图片地址
- 只有 `web` / `devtools` 环境才允许直接用远程 URL 兜底
- 目的：
  - 避免 WebView 直接加载自签名 HTTPS 图片时触发 `chromium net_error -101`

### 6. 攻略详情页头像与评论头像改成走缓存图片组件

文件：`pages/guide-detail/index.vue`

## 这次排查日志时看到的关键信息

### 1. 老问题的根日志

日志里反复出现：

- `Failed to connect to /111.20.31.227:34144`
- `ECONNREFUSED`
- `chromium ... ssl_client_socket_impl ... net_error -101`

这说明当时存在两类问题：

- 仍有旧资源地址没有被替换掉
- WebView 仍有部分链路在直接请求自签名 HTTPS 资源

### 2. 直接抓接口返回后确认了“后端仍返回旧图地址”

通过直接请求：

- `https://yd.frp-arm.com:44637/api/guides`

拿到的响应中，至少一条攻略仍返回：

- `"image":"https://111.20.31.227:34144/uploads/guides/..."`

这证明：

- 前端做旧地址兼容改写是必要的
- 不能假设后端已经把所有历史数据都修干净

### 3. 为什么“现在模拟器里可以显示内容”

这是合理现象，不矛盾。

因为页面内容可能来自不同链路：

- 有些图片本来就是公网正常地址，例如 Wikimedia
- 有些图片已经被下载并缓存到本地文件
- 有些接口数据经过前端改写后已可正常显示
- 真正出问题的只是少数仍带旧地址或仍直连 HTTPS 的资源

## 仍需注意的事项

- 后端历史数据里可能仍残留旧资源地址，建议后端后续统一清洗
- 攻略详情中的视频资源仍然是相对更敏感的一条链路，如果后续再出现 HTTPS 问题，优先检查：
  - `guide.video`
  - `guide.videoPoster`
- AMap 仍可能有 `USERKEY_PLAT_NOMATCH` 这类授权问题，但它不是这次攻略页显示问题的主因
- `CachedImage` 目前已经更偏向“正式远程地址直接显示 + 本地缓存补充”，后续如果更换域名或证书，需要一起复核这条策略

## 这次最关键的经验

- 不能只修接口域名，还要修接口返回的数据内容
- 不能只修请求，还要修图片、头像、视频等资源加载方式
- 离线 uni-app Android 包要确认真正运行的是同步后的 `app-service.js`，不能只改源码不改离线资源
- 真机表现和模拟器经常不同，尤其是头像、本地缓存路径、WebView 图片加载策略，必须以真机验证为准
